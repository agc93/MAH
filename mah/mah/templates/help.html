Introduction

You receive a phone call from someone who claims to work with you and asks you about information that isn't public. How can you be sure that they are who they say they are and that they really do work for your company?

This system, MAH Verification, is designed to help. It enables you to verify that the person you are communicating with via email, telephone or IRC is the person they say they are and also company associate.

It uses 2 factor authentication (PIN + token) to provide the strongest and most practical assurance that someone is a current company associate. It allows you to create a short authentication code for another associate. This authentication code is a constant or interval character hex-string, consisting of letters from a-f and numbers 0-9. By default, codes have a short lifetime (10 minutes).

The Process of Authentication

The system performs a one way authentication via a authentication code, as illustrated in the following example:

  1. Bob and Mary are based at different locations and are both members of a large project team.
  2. The project is confidential and they are under strict instructions not to discuss any details with anyone external to the project team.
  3. Bob calls Mary to ask some questions about the work Mary is doing on the project.
  4. Mary doesn't usually speak to Bob on the phone and doesn't recognise his voice.
  5. Mary logs in to the verification system using her username and her PIN + token code.
  6. Mary searches for Bob and generates an authentication code for him.
  7. Mary says "Bob, before I discuss this information over the phone, I'd like to verify your identity, so I have authenticated you. Can you please read back the authentication code for me?"
  8. Bob also logs into the verification system and sees the authentication from Mary and reads back the authentication code for Mary.
At this point, Mary has authenticated Bob, but Bob has not authenticated Mary. If Bob wants to do that, he can create a new and different code that Mary will need to read back to him.

Searching

Search terms must be longer than 2 characters and only contain alphanumeric characters (a-z A-Z 0-9). Searches are not case sensitive and peformed on the username (uid), full name and phone number fields of the staff directory. Currently, only the desk phone field is searched, not the home phone or mobile fields.

Wildcards and regular expressions are not accepted in search fields, but search terms are automatically wildcarded. For example, the search term "mark" would match users with a first name "Mark" or "Marko", or with a surname of "Denemark".

Reporting Suspicious Authentications

If you attempt to authenticate someone and they are unable to complete it or they repeatedly fail to read back the correct authentication code, this may be cause for concern that "something isn't right". If you feel that there is something suspicious or strange about your interaction, then please report it.

You can also report suspicious interactions via the "flag a suspicious authentication" link in the top right hand corner of the page. Using this option, you will be prompted to select the specific authentication from a history of past authentications.

You can also report a specific authentication as suspicious by clicking on the "flag as suspicious" link after that authentication. In all cases, there is a text box to provide more information, please use this to describe why this authentication was being reported. Please provide as much relevant information as you can.

Hints

Phonetic (NATO) hints for authentication codes

By hovering your mouse over the authentication code, you can see the code converted to the phonetic (NATO) alphabet. By using this, when reading out codes, it makes it easier to hear and understand codes, with less chance of confusion.
